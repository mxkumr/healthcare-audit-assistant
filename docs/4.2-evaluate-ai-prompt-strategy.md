## What we're doing

Evaluate AI answers natural-language questions **on the current ALP screen** by sending a **compressed CAP context window** to SAP AI Core (GPT-5 deployment). Hallucinations are limited by:

1. Separating **persona/rules** (system) from **schema + data** (user JSON)  
2. Trusting **precomputed CDS columns** (no invented formulas)  
3. Requiring evidence only from **`dataSnapshot.rows`**

---

## Prompt topology (context window)

```
┌─────────────────────────────────────────────────────────────┐
│ SYSTEM — Lead Medicare Auditor + Agency framework           │
│ (no raw rows; anti-hallucination + Task 2 tier protocol)    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ USER — single compressed JSON payload from CAP              │
│                                                             │
│  {                                                          │
│    schemaSnapshot: { alp, columns, rankingHints,            │
│                      task2ClassificationTiers?, ... },      │
│    dataSnapshot:   { entity, rowCount, truncated, rows[] }, │
│    question:       "<auditor question>"                     │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    SAP AI Core chat/completions
                    (temperature / frequency_penalty /
                     presence_penalty only)
```

### Why this layout minimizes hallucinations

| Layer | Role |
|-------|------|
| **System persona** | Forces Lead Auditor behavior and a fixed 4-step agency protocol |
| **`schemaSnapshot`** | Tells the model *which ALP*, *which columns*, *how to rank* — without free-form invented thresholds |
| **`dataSnapshot.rows`** | Only allowed facts; identifiers/amounts must be quoted from these rows |
| **Task 2 tiers** (when present) | Model must reason with CAP-stamped `EfficiencyCategory` / `UtilizationCategory` *before* ranking dollars |
| **Structured Markdown output** | Finding → Evidence → Confidence → Follow-up (easier to audit) |

### Agency framework (always in order)

1. **ALP frame** — read `schemaSnapshot.alp` + `alpGuidance`; pick ranking columns via `rankingHints`  
2. **Task 2 classification** — if `task2ClassificationTiers` exists, filter/interpret using Efficiency × Utilization bands on each row  
3. **Evidence scan** — scan **all** `dataSnapshot.rows`; aggregate / top-N / counter-examples  
4. **Actionability** — one concrete follow-up on this ALP

```json
  "alp": {

    "task": "2.1",

    "grain": "Year × NPI"
  },
  "columns": ["Year", "NPI", "ProviderName", "..."],
  "primaryMetrics": ["CostPerBeneficiary", "EfficiencyCategory", "..."],
  "rankingHints": {
    "highCostOutliers": "filter EfficiencyCategory=High-Cost Outlier then rank CostPerBeneficiary"
  },
  "task2ClassificationTiers": {
    "EfficiencyCategory": ["Highly Efficient", "Average Spend", "High-Cost Outlier"],
    "UtilizationCategory": ["Low Utilization", "Moderate Utilization", "High Utilization"],
    "authority": "Task 2 CAP classifications already stamped on each row..."
  },
  "alpGuidance": ["..."],
  "antiHallucination": ["Use ONLY rows in dataSnapshot.rows", "..."]
}
```

### Data snapshot layout

Built by `buildDataSnapshot()` after `SELECT` from the ALP’s CDS entity (max **400** rows):

```json
{
  "source": "CAP MedicareService",
  "entity": "ProviderCostEfficiency",
  "format": "json-rows",
  "rowCount": 400,
  "truncated": true,
  "columns": ["Year", "NPI", "..."],
  "rows": [ { "Year": "2022", "NPI": "...", "EfficiencyCategory": "High-Cost Outlier", "...": "..." } ]
}
```

### AI persona & Task 2 reasoning

- **Persona:** Lead Medicare Auditor (Fiori ALP explanation agent)  
- **Task 2 link:** On 2.1 / 2.2b, `task2ClassificationTiers` is injected so the model must treat CAP classification labels as authoritative before attributing spend to inefficiency vs volume  

### AI Core request parameters

Only:

```json
{ "temperature": 0, "frequency_penalty": 0, "presence_penalty": 0 }
```

(No `max_tokens` / `stop` — required for GPT-5 deployment `df07e2232bf39cb5`.)

Deployment URL is set via CF env `AI_DEPLOYMENT_URL` (see `docs/joule.json` → `aiDeployments`).

### Expected answer shape

Natural audit memo (not system jargon):

```markdown
## Finding
Urban / Metro carries the largest rejected-charge burden in this extract…

## Evidence
- Procedure 99214 in Urban / Metro: rejected charges about $40.7M on ~306k services
- Suburban and Rural peaks are far smaller (e.g. Suburban 99214 ~$2.9M)

## Confidence
Medium — the extract supports the tier ranking, but I only listed a few verified top codes.

## Follow-up
Filter to Urban / Metro and sort by rejected charges descending to review the largest procedure drivers.
```

Avoid phrases like `ALP frame`, `dataSnapshot.rows`, `truncated: false`, or raw JSON keys in the visible answer.

### What “good” looks like

- Natural Lead Auditor tone  
- Correct ranking column for the question  
- Few **accurate** cited figures (better than many invented ones)  
- On 2.1/2.2b, Task 2 category labels before dollar rankings  
- Confidence and Follow-up readable by a non-developer auditor
